# 「 🌐 Open Source & Community Contribution Guide 」

> *"Open source is how developers grow faster — by reading great code, contributing fixes, and building in public."*

---

## ⚡ 1. Philosophy & Licensing

I believe in accessible, well-documented software:
* **MIT License Default:** All developer tools, utility libraries, and template projects in this portfolio are released under the open MIT License.
* **Build in Public:** I share project structures, coding patterns, and architecture decisions openly to help other developers learn.

---

## 🤝 2. Pull Request (PR) Guidelines

If you are contributing to any of my public repositories, please adhere to the following workflow:
1. **Fork & Branch:** Create a feature branch from `main` using descriptive naming (e.g., `feature/user-auth` or `fix/api-cors-error`).
2. **Clean Commits:** Write clear, concise commit messages explaining *why* a change was made, not just what was changed.
3. **Linting & Formatting:** Ensure code follows ESLint rules (for JS/React) and Prettier for consistent formatting.
4. **Documentation:** Any new route, component, or environment variable must be documented in the repository's `README.md`.

---

## 📬 3. Proposing New Features

Have an idea for a new feature or improvement?
* Open an **Issue** with the `[FEATURE]` tag detailing the use case, expected behavior, and proposed implementation approach.
* Reach out directly via the channels listed in [COLLAB.md](./COLLAB.md) for co-developing larger features or complete projects.
